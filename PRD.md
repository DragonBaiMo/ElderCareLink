# 《基于 SpringBoot 的关爱空巢老人管理系统》

产品需求文档（PRD）

---

## 一、项目名称（中文）

**项目名称**：基于 SpringBoot 的关爱空巢老人管理系统

---

## 二、系统定位

本系统面向社区及基层组织的**空巢老人关爱工作场景**，通过信息化手段整合老人基本信息、健康记录、志愿者探访及社区公告等数据，为管理员、志愿者与家属提供统一的数字化管理平台。

系统性质为：

* **本科毕业设计演示系统**
* 架构采用单体 Web 应用，前后端分离或简易整合即可
* 目标是实现“**核心功能完整、界面清晰、能顺利演示**”，不追求商用级性能与复杂架构

---

## 三、用户角色

系统主要面向以下三类角色：

### 3.1 系统管理员

**典型身份**：社区工作人员、居委会干事等。

**主要职责与使用需求：**

* 管理系统用户及角色权限
* 维护老人基本信息及健康档案
* 管理志愿者信息与探访记录
* 发布社区公告与活动通知
* 查看数据统计与报表，用于工作汇报和情况掌握

### 3.2 志愿者

**典型身份**：社区志愿者、社会组织成员等。

**主要职责与使用需求：**

* 登录系统查看自己负责的老人列表
* 记录探访情况（时间、内容、老人状态、后续建议等）
* 查看与自己相关的公告与活动通知
* 在需要时补录或修改本人的历史探访记录（权限受控）

### 3.3 家属用户（可选角色）

**典型身份**：空巢老人的子女或其他亲属。

**主要使用需求：**

* 登录后查看绑定老人（自家长辈）的基本信息
* 查看老人近期的健康记录与志愿者探访情况
* 浏览与老人相关的社区活动、公告信息

> 家属角色相关功能可视为 **Should/Could 级**，根据开发时间与工作量灵活实现。

### 3.4 未登录用户

* 不允许访问系统内部业务页面
* 可选提供简单登录页面与系统简介页面

## 四、功能场景（业务使用场景）

以下为系统围绕各角色设计的关键使用场景，用于答辩展示和需求说明。

### 场景 1：管理员建立老人档案

1. 管理员登录系统
2. 进入「老人信息管理」模块
3. 点击“新增老人”，录入姓名、性别、年龄、住址、居住类型、健康等级、紧急联系人等信息
4. 保存后，老人自动出现在老人列表中
5. 管理员可按姓名、居住类型等条件查询老人档案

**目的**：完成对空巢老人的基础信息建档，为后续健康记录与探访记录提供基础数据。

---

### 场景 2：志愿者上门探访并记录服务情况

1. 志愿者登录系统
2. 在「我的探访任务」（或老人列表）中查看自己负责的老人名单
3. 实际上门或电话沟通后，回到系统
4. 在「探访记录」模块中选择对应老人并填写探访时间、探访方式、服务内容、老人状态与后续建议
5. 保存记录，管理员可在后台查看该志愿者的服务记录

**目的**：记录志愿服务过程，形成可追溯的关爱档案。

---

### 场景 3：管理员查看整体运行情况并导出报表

1. 管理员登录系统
2. 打开「数据统计与报表」模块
3. 查看老人数量、健康等级分布、志愿者探访次数等图表统计
4. 可将老人基本信息或统计结果导出为 Excel，便于线下汇报或归档

**目的**：为社区掌握空巢老人总体情况提供直观数据支持。

---

### 场景 4：发布社区公告与活动通知

1. 管理员登录系统
2. 进入「公告管理」模块
3. 新增公告，填写标题、内容、公告类型（活动通知、政策信息等）、发布时间
4. 设置公告状态为“已发布”
5. 志愿者和家属用户登录后可在「公告列表」中查看相关内容

**目的**：作为社区与志愿者、家属之间的信息发布渠道。

---

### 场景 5：家属查看老人情况（可选）

1. 家属使用系统提供的账号登录
2. 系统根据账号绑定关系展示对应老人的基本信息
3. 查看老人近期的健康记录列表和志愿者探访记录摘要
4. 了解老人近期健康状况与社会关爱情况

**目的**：为子女提供信息透明的窗口，缓解对远方父母情况的焦虑。

---

## 六、项目边界（明确不做什么）

为确保系统在毕业设计时间内可完成且可稳定演示，本项目限定如下边界：

1. **不涉及支付功能**

   * 不对接任何第三方支付平台
   * 不实现线上费用结算、补贴发放等功能

2. **不进行医疗诊断与智能评估**

   * 系统仅做健康数据记录与简单展示
   * 不做医学诊断结论，不提供自动健康建议或处方
   * 不接入复杂的算法模型与大数据分析

3. **不实现即时通讯与音视频功能**

   * 不实现实时聊天、视频通话等功能
   * 不对接第三方消息推送平台（如短信、公众号推送等），仅在站内展示公告和信息

4. **不建设复杂的多组织、多社区多租户架构**

   * 系统默认面向单个社区或单一管理单位
   * 不支持跨社区的数据隔离与多租户管理

5. **不考虑高并发与分布式部署**

   * 不拆分为微服务或云原生架构
   * 仅支持在一台普通计算机上以单体应用方式运行

6. **不实现复杂权限体系**

   * 仅按角色（管理员 / 志愿者 / 家属）进行基础权限控制
   * 不做细粒度的字段级权限与数据隔离

7. **不实现复杂日志审计与安全审计功能**

   * 可保留基础的操作记录（如创建时间），不做完整安全审计系统

## 七、验收标准（用于答辩展示）

为保证系统在答辩中可顺利展示并通过评审，制定如下验收标准：

### 7.1 功能实现验收

1. **Must 级功能全部可用**

   * 管理员、志愿者至少两类角色可以正常登录
   * 可以完成老人信息的新增、查询和修改
   * 可以录入并查询健康记录
   * 可以录入并查询志愿者探访记录
   * 可以发布公告并在前台公告列表中看到

2. **核心业务闭环可完整演示**

   * 场景 1：管理员新增老人档案 → 查看老人列表
   * 场景 2：志愿者登录 → 记录探访 → 管理员查看探访记录
   * 场景 3：管理员查看数据统计（至少一个统计图表或统计数值）
   * 场景 4：管理员发布公告 → 其他角色登录查看公告

### 7.2 界面与交互验收

1. 系统界面语言统一为**简体中文**，菜单、按钮命名规范清晰
2. 页面布局合理，常用操作入口清晰，对本科毕设评委具有良好可读性
3. 对危险操作（如删除记录）有二次确认提示或明显标识

### 7.3 数据与稳定性验收

1. 在本地环境下，连续执行基础查询、录入、修改操作不会出现系统崩溃或严重错误
2. 数据库结构清晰，提供完整建表脚本，能在新环境中重建并运行系统
3. 核心数据表（老人、健康记录、探访记录、公告等）结构合理，能支撑主要业务场景

### 7.4 非功能要求验收

1. 登录密码采用加密存储方式（代码中可见相应实现）
2. 常规操作（如列表查询）在本地环境下响应时间合理（一般在 3 秒内）
3. 提供简要的异常处理机制，对于访问不存在页面或错误操作给出友好提示

### 7.5 文档与说明验收

1. 提供完整的：

   * 开题报告
   * 本 PRD 文档
   * PDR（程序设计说明书）
   * IPO（开发流程说明）
2. 提供《系统运行说明》或在论文中明确：

   * 环境配置步骤
   * 数据库创建与初始化方法
   * 系统启动顺序与访问地址
3. 论文中对系统功能、技术路线与实现过程有清晰叙述，图文对应，结构完整。
