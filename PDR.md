# 《基于 SpringBoot 的关爱空巢老人管理系统》

程序设计说明书（PDR）

---

## 一、项目概述

### 1.1 项目背景

在我国人口老龄化加剧、空巢与独居老人比例上升的背景下，单一的物质养老已无法满足老年群体的心理慰藉与社会关怀需求。大量研究表明，孤独感、抑郁、焦虑与社会支持不足密切相关，社区、家庭与志愿者之间缺乏有效协同与信息共享机制。

本项目拟设计并实现一个**基于 SpringBoot 的关爱空巢老人管理系统**，以社区为应用场景，通过信息化手段整合老人信息、健康记录、志愿探访与消息公告，为管理员、志愿者与家属提供统一的数字化管理与关爱平台。系统定位于**本科毕业设计演示级别**，以“能跑、能演示、结构清晰”为核心目标。

### 1.2 建设目标

1. 建立一套简单清晰的**空巢老人关爱信息管理平台**，实现老人基本信息、健康记录与探访记录的统一管理。
2. 提供面向管理员、志愿者、家属的**多角色操作界面**，支撑基本的查询、录入与反馈操作。
3. 通过数据统计与报表，实现对老人数量、服务频次、健康记录等的**可视化展示**，方便社区进行整体掌握与汇报。
4. 在技术上采用 **SpringBoot + MySQL + 简单前端（如 Vue）** 的经典单体架构，保证普通本科生在一个学期内可以完成开发与演示。

### 1.3 设计原则

* **毕业设计优先**：以文档规范和功能可演示为第一目标，不追求复杂架构与高并发能力。
* **功能适度**：功能划分明确，核心功能完整可用，扩展功能点到为止，避免过度设计。
* **实现可行**：所有模块均保证在本地单机环境下可运行，开发难度适配本科生水平。
* **数据真实感**：通过合理的数据模型与样例数据，让系统演示尽量贴近真实社区场景。

## 二、模块清单与功能范围

本系统采用模块化设计，将功能划分为若干相对独立的子模块。各模块既可单独开发调试，又能通过统一的权限与菜单体系进行整合。

### 2.1 用户与权限管理模块

**功能定位**：维护系统登录账号及角色，控制不同用户访问的菜单和功能范围。

**主要功能：**

* 用户登录 / 注销
* 角色管理（管理员 / 志愿者 / 家属）
* 用户基本信息维护（账号、姓名、联系方式、角色分配）
* 登录密码修改

### 2.2 老人信息管理模块

**功能定位**：作为系统的基础数据模块，记录空巢老人的基本信息、健康概况和紧急联系信息。

**主要功能：**

* 老人基本信息录入（姓名、性别、年龄、联系方式、住址、居住类型等）
* 健康概况记录（慢性病情况、基础健康等级等）
* 紧急联系人信息维护（姓名、电话、与老人的关系）
* 老人信息查询与条件检索（按姓名、居住类型、健康等级等）
* 老人信息修改与注销（软删除标记）

### 2.3 健康信息记录与查询模块

**功能定位**：对老人的关键健康指标进行记录与查询，为后续健康趋势分析提供依据。

**主要功能：**

* 日常健康记录录入（血压、心率、体温、血糖等基础指标）
* 健康记录列表查看（按老人、按日期查询）
* 关键指标简单标记（如超出预设范围以红色标注）
* 健康记录的增删改查
* 简单的趋势查看（如按日期排序的列表，可选折线图展示）

### 2.4 志愿者管理与探访记录模块

**功能定位**：记录志愿者信息及其对老人的探访情况，反映服务频次与服务内容。

**主要功能：**

* 志愿者基本信息维护（姓名、联系方式、所属组织等）
* 探访计划登记（可选，简化为预定探访时间与对象）
* 探访记录录入（探访时间、探访对象、服务内容、老人当时状态、建议等）
* 探访记录查询（按志愿者、按老人、按时间段）
* 志愿服务统计（简单统计某时间段内探访次数）

### 2.5 社区公告与消息通知模块

**功能定位**：发布与管理社区关爱活动、健康讲座、政策通知等信息，实现对志愿者和家属的统一告知。

**主要功能：**

* 公告信息发布（标题、内容、发布人、发布时间）
* 公告列表浏览与详情查看
* 按发布时间、发布人、公告类型进行筛选
* 公告状态控制（草稿、已发布、下架）

（消息是否已读、站内通知可视为扩展功能）

### 2.6 数据统计与报表模块

**功能定位**：以图表或简单报表形式展示系统核心数据，辅助社区进行管理决策和成果展示。

**主要功能：**

* 老人数量统计（按年龄段、健康等级、居住类型等维度）
* 志愿服务统计（探访总次数、志愿者服务次数排行等）
* 简单健康记录统计（如高血压老人数量、最近一周录入记录数量）
* 基本统计报表导出（Excel / PDF，可选择实现其一）

### 2.7 系统配置与基础数据模块

**功能定位**：维护系统中的部分枚举与基础参数，支持后期调整而无需修改代码。

**主要功能：**

* 居住类型字典维护（如空巢、独居、与子女同住但子女常年外出等）
* 健康等级字典维护（良好、一般、重点关注等）
* 健康指标阈值设置（如血压、心率等简单上下限，用于前端标红）

## 三、技术架构（简单、可落地、本地能跑）

### 3.1 整体架构

系统采用典型的**单体 Web 应用架构**，前后端分离方式实现，适配本科生开发与演示：

* 表现层：基于浏览器的 Web 前端（推荐 Vue 框架 + Element UI 等组件库）
* 业务层：SpringBoot 构建的 REST 风格后端服务
* 数据持久层：Spring Data JPA 或 MyBatis + MySQL 数据库
* 部署方式：本地单机运行（以 Windows 环境为主），通过 IDEA 直接启动后端，通过 Node/npm 启动前端开发服务或打包成静态资源由后端统一提供

### 3.2 关键技术选型

* 开发语言：Java 8/11 及以上
* 后端框架：

  * `org.springframework.boot:spring-boot-starter-web`
  * `org.springframework.boot:spring-boot-starter-data-jpa` 或 `org.mybatis.spring.boot:mybatis-spring-boot-starter`
  * `org.springframework.boot:spring-boot-starter-validation`
* 前端技术：Vue2/Vue3（二选一即可）+ Axios + 简单组件库（如 Element UI / Element Plus）
* 数据库：MySQL 5.7 / 8.0
* 构建工具：Maven
* 开发工具：IntelliJ IDEA、Node.js、Chrome 浏览器
* 图表展示（可选）：ECharts 或类似前端图表库

### 3.3 系统逻辑架构分层

* 表现层（Controller / 前端页面）：接收请求、返回 JSON 或视图、处理基本参数校验。
* 业务层（Service）：封装业务逻辑，如健康记录录入校验、探访记录归档等。
* 持久层（Repository / Mapper）：负责与数据库交互，实现增删改查。
* 数据层（Database）：存储核心业务数据，以范式化设计为主。

### 3.4 部署与运行方式（毕业设计级）

* 数据库：在本地安装 MySQL，创建专用数据库，并导入建表脚本。
* 后端：在 IDEA 中导入 Maven 项目，配置 `application.yml` 中的数据库连接信息，直接运行 SpringBoot 主类即可。
* 前端：通过 Node/npm 安装依赖，使用开发命令启动前端调试服务，或打包后放入 SpringBoot 的静态资源目录中统一访问。
* 访问方式：浏览器访问 `http://localhost:8080`（或前端实际端口），使用预置账号登录演示。

## 四、数据库概要设计

本系统采用关系型数据库 MySQL，围绕“老人、健康记录、志愿者、探访记录、公告、用户”六类核心实体展开设计。以下为主要数据表及其关键字段概述（仅为概要，具体字段可在后续详细设计中补充和优化）。

### 4.1 用户与角色相关

1. **t_user（用户表）**

   * 主要字段：

     * `id`：主键
     * `username`：登录账号
     * `password`：登录密码（加密存储）
     * `real_name`：真实姓名
     * `phone`：联系电话
     * `role`：角色标识（管理员 / 志愿者 / 家属）
     * `status`：账号状态（正常 / 禁用）
     * `create_time`：创建时间

2. **t_volunteer_profile（志愿者扩展信息表，可选）**

   * 主要字段：

     * `id`：主键
     * `user_id`：关联 t_user
     * `organization`：所属组织
     * `remark`：备注

（如开发时间有限，可将志愿者信息直接合并在 t_user 中，通过角色区分）

### 4.2 老人与健康信息相关

1. **t_elder（老人信息表）**

   * 主要字段：

     * `id`：主键
     * `name`：姓名
     * `gender`：性别
     * `age`：年龄
     * `id_card`：身份证号（可选）
     * `phone`：联系电话
     * `address`：详细住址
     * `living_type`：居住类型（空巢、独居等，对应字典表）
     * `health_level`：健康等级（良好、一般、重点关注等）
     * `emergency_contact_name`：紧急联系人姓名
     * `emergency_contact_phone`：紧急联系人电话
     * `emergency_contact_relation`：与老人的关系
     * `remarks`：备注
     * `status`：状态（在册 / 注销）

2. **t_health_record（健康记录表）**

   * 主要字段：

     * `id`：主键
     * `elder_id`：关联 t_elder
     * `record_date`：记录日期时间
     * `blood_pressure`：血压（可用字符串如“120/80”）
     * `heart_rate`：心率
     * `temperature`：体温
     * `blood_sugar`：血糖（可选）
     * `health_desc`：健康情况说明
     * `created_by`：录入人（关联 t_user）
     * `create_time`：录入时间

### 4.3 志愿服务与探访记录相关

1. **t_visit_record（探访记录表）**

   * 主要字段：

     * `id`：主键
     * `elder_id`：探访老人（关联 t_elder）
     * `volunteer_id`：志愿者（关联 t_user 或 t_volunteer_profile）
     * `visit_time`：探访时间
     * `visit_type`：探访类型（入户探访、电话沟通等）
     * `service_content`：服务内容描述
     * `elder_status`：老人当时状态简述
     * `next_advice`：后续建议
     * `create_time`：记录时间

### 4.4 公告与消息相关

1. **t_announcement（公告表）**

   * 主要字段：

     * `id`：主键
     * `title`：公告标题
     * `content`：公告内容
     * `publisher_id`：发布人（关联 t_user）
     * `publish_time`：发布时间
     * `status`：状态（草稿 / 已发布 / 下架）
     * `type`：公告类型（活动通知、政策信息等，可选）

2. **t_announcement_read（公告阅读情况表，可选）**

   * 主要字段：

     * `id`：主键
     * `announcement_id`：关联公告
     * `user_id`：阅读人
     * `read_time`：阅读时间

### 4.5 字典与配置相关（可合并设计）

1. **t_dict（数据字典表）**

   * 主要字段：

     * `id`：主键
     * `dict_type`：字典类型（如 living_type、health_level 等）
     * `dict_key`：键值
     * `dict_value`：显示名称
     * `sort`：排序值
     * `status`：状态

2. **t_system_config（系统配置表，可选）**

   * 用于存储健康阈值等参数，字段可根据实际需要精简设计。

## 五、功能优先级规划（Must / Should / Could）

### 5.1 优先级定义

* **Must**：答辩演示时必须可运行、可操作的核心功能，保证系统主线闭环。
* **Should**：建议实现的功能，用于提升系统实用性与完整度，但个别细节可简化。
* **Could**：在时间与精力允许的前提下可选实现的拓展功能，用于加分展示。

### 5.2 功能优先级列表

（表格为逻辑结构，实际可在论文或说明书中以 Word 表格呈现）

1. **Must 级功能（必须实现）**

* 用户登录 / 注销
* 基于角色的菜单展示（至少区分管理员、志愿者）
* 老人信息的新增、查询、修改、注销
* 健康记录的新增、查询与列表展示
* 志愿者探访记录的新增与查询
* 公告发布与公告列表查看
* 基本数据统计：

  * 老人总人数
  * 按健康等级统计人数
  * 探访记录总数量

2. **Should 级功能（建议实现）**

* 老人信息的条件查询与分页显示
* 健康记录按时间范围查询
* 志愿者服务次数统计（按志愿者汇总探访次数）
* 简单图表展示（如老人健康等级饼图、探访次数柱状图）
* 公告状态管理（草稿 / 已发布 / 下架）
* 数据字典配置界面（居住类型、健康等级等的前端维护）
* 简单数据导出（如将老人基本信息导出为 Excel）

3. **Could 级功能（可选实现）**

* 健康指标超阈值的简单预警标识（如列表中标红或弹出提示）
* 公告已读记录与“未读公告”提醒（界面提示为主，不必接入真实推送）
* 家属角色账号登录后查看绑定老人的基本信息与探访记录
* 简单的地图位置展示（可仅用静态地图或文字描述，不强制接入地图服务）
* 健康趋势图（按某位老人的血压/体温记录绘制折线图）

## 六、非功能需求（毕业设计级别）

### 6.1 性能需求

* 系统主要运行于本地单机环境，支持**同时在线用户不超过 20 人**的操作场景。
* 在本地网络环境下，常规查询与列表加载的响应时间尽量控制在 **3 秒以内**。
* 无需考虑高并发、分布式事务等复杂场景。

### 6.2 安全性需求

* 提供基本的登录认证机制，未登录用户不得访问系统业务页面。
* 用户密码采用不可逆加密存储（如使用 MD5 / BCrypt 等，任选一种即可）。
* 对表单输入进行基本校验，避免出现明显的 SQL 注入与脚本注入问题。
* 不涉及真实支付，不保存敏感金融信息。

### 6.3 易用性需求

* 系统界面语言采用**简体中文**，操作按钮与菜单命名清晰明了。
* 页面布局简洁，常用功能通过左侧菜单或顶部导航直接进入。
* 重要操作（如删除、注销）提供确认提示，减少误操作。

### 6.4 可维护性需求

* 代码按照 Controller、Service、Repository 等分层组织，命名规范统一。
* 关键业务逻辑编写必要注释，便于后续维护与答辩时讲解。
* 配置信息（数据库连接、端口、阈值等）集中在配置文件中管理，避免硬编码。

### 6.5 可靠性与数据备份

* 对常见异常进行统一处理，给出友好提示页面或提示信息。
* 在开发阶段通过 JUnit 或简单手工测试验证核心业务流程的稳定性。
* 提供数据库建表脚本与示例数据，以便在不同环境中快速恢复和演示。

### 6.6 演示与答辩支持

* 预置至少一组管理员账号和若干志愿者账号，用于现场演示不同角色的功能。
* 在数据库中预先插入一定数量的老人信息、健康记录与探访记录，保证图表和统计模块演示有数据支撑。
* 准备好系统运行步骤说明（启动数据库 → 启动后端 → 启动前端 → 浏览器访问），便于答辩时快速切换与展示。
